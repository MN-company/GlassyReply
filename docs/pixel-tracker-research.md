# Pixel Tracker Research Notes

## Current reality
- Gmail still fetches remote images through its own proxy, so image requests still happen, but the fetch metadata belongs to Google and not to the human browser session.
- Apple Mail Privacy Protection still preloads remote content in the background, so image opens cannot be trusted as a human-open signal in that client family.
- CSS `background-image` remains a useful secondary beacon because support is broad enough across modern email clients to provide another signal path beyond a naked 1x1 `<img>`.
- `@font-face` remains niche in email clients, so it should be treated as experimental diagnostic telemetry and never as the primary open signal.

## Multilayer strategy
1. Use a signed primary image beacon with a non-suspicious `2x1` asset path, not a raw `1x1` query-string pixel.
2. Add a CSS `background-image` beacon on a tiny low-opacity element for clients that still resolve CSS remote assets.
3. Add a dark-mode CSS beacon so WebKit-like renderers can expose an extra signal when the dark-mode media query evaluates.
4. Add a tiny remote font beacon as diagnostic-only telemetry. It helps identify clients that resolve font assets, but it must not count as a real open by itself.
5. Classify events server-side into `human_browser`, `gmail_proxy`, `prefetch_proxy`, `font_loader`, or `unknown_proxy`.
6. Surface confidence and layer in Telegram instead of collapsing every hit into a binary “opened / not opened”.

## Real tests run in this repo
- Local Cloudflare Worker dev server via `wrangler dev`.
- Independent webhook capture server via `python3 scripts/pixel_smoke_test.py capture`.
- Gmail-proxy simulation with `curl -A 'GoogleImageProxy' -H 'X-Gmail-Fetch-Info: msgid=probe'`.
- Real Chromium render via `playwright screenshot` against a standalone HTML fixture generated by `python3 scripts/pixel_smoke_test.py bundle`.
